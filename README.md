Factory Method

Проблема

- Коли код створює об’єкти напряму через new, він стає жорстко прив’язаним до конкретних класів.
Це ускладнює розширення та зміну логіки створення об’єктів.

Ідея

- Винести створення об’єкта у окремий метод, який перевизначається підкласами.

UML
        
       Creator
           |
    ----------------------
    |                    |
    ConcreteCreatorA  ConcreteCreatorB
           |
        Product
           |
    ----------------------
    |                     |
    ConcreteProductA  ConcreteProductB

Де застосовують

- логістичні системи

- платіжні сервіси (створення різних способів оплати)

- UI-фреймворки


Adapter

Проблема

- Є несумісні інтерфейси, які потрібно змусити працювати разом.

Ідея

- Створити "перехідник", який перетворює один інтерфейс в інший.

UML
-       Client → Target
                    ↑
                Adapter
                    |
                Adaptee
Де застосовують
- інтеграція старих API
- підключення сторонніх бібліотек
- робота з legacy-системами


Observer

Проблема
- Потрібно автоматично повідомляти багато об’єктів про зміну стану іншого об’єкта.

Ідея
- Створити механізм підписки/повідомлення.

UML
-       Subject
          | attach()
          | detach()
          | notify()
              ↓
            Observer
                ↑
        ConcreteObserver
Де застосовують
- системи сповіщень
- чат-додатки
- GUI
- біржові системи
- реактивні фреймворки
